# UX-07 — บน iPhone จริงแถบเมนูล่างโดน home indicator ทับ + เราเทสจอเดียว

Status: ready-for-human
Owner: A
Priority: H — คีนเจอเองบนเครื่องจริง · กระทบทุกคนที่เปิดบน iPhone รุ่น X ขึ้นไป
Refs: src/app/layout.tsx, src/app/globals.css, src/components/app-nav.tsx, playwright.config.ts

## 1. บั๊กจริง: `env(safe-area-inset-*)` คืนค่า 0 เสมอ

`app-nav.tsx:15` เขียน `pb-[env(safe-area-inset-bottom)]` ไว้แล้ว — **เจตนาถูก**
แต่ `env(safe-area-inset-*)` จะมีค่าก็ต่อเมื่อหน้าประกาศ `viewport-fit=cover`
ซึ่งเราไม่ได้ตั้ง (ยืนยันจาก HTML จริง: `width=device-width, initial-scale=1` เท่านั้น)

→ บน iPhone X ขึ้นไป แถบเมนูล่างนั่งชิดขอบจอ แล้วแถบ home indicator ของ iOS
ทับพื้นที่กดลงไปอีกราว 34px · ปุ่มล่างสุดโดนบีบและกดยาก

**เทสของเราไม่มีทางจับได้** — Playwright ใช้ Chromium ซึ่งไม่มี home indicator
โค้ดจึงดูเหมือน "จัดการแล้ว" ทั้งที่ไม่มีผลอะไรเลย

## 2. เราเทสความกว้างเดียว

`playwright.config.ts` ใช้ `devices["iPhone 14"]` = 390px อย่างเดียว

iPhone จริงแคบกว่านั้นได้:

| รุ่น / โหมด | ความกว้าง |
| --- | --- |
| iPhone SE (รุ่นแรก) | 320 |
| iPhone SE 2/3 · 12 mini | 375 |
| **iPhone รุ่นใดก็ได้ที่เปิด Display Zoom** | **320** |

Display Zoom คือตัวที่คนมองข้าม — เปิดใน Settings → Display & Brightness แล้ว
iPhone 15 จะรายงานความกว้างเป็น 320px ทั้งที่เครื่องใหญ่เท่าเดิม

ที่ 320px เนื้อหาเหลือ 288px หลังหัก `px-4` สองข้าง = **แคบกว่าที่ออกแบบไว้ 20%**

## ตรวจแล้วไม่พัง (สำคัญ — ปัญหาไม่ใช่ layout แตก)

ไล่ทั้ง 7 หน้า × 4 ความกว้าง (320/360/375/390): ไม่มี h-scroll · ไม่มี element ล้นกรอบ ·
ไม่มีปุ่มเตี้ยกว่า 44px แม้แต่จุดเดียว → ความรู้สึก "บีบ" มาจากพื้นที่หายไป 20% ไม่ใช่ของแตก

## ตรวจแล้วดีอยู่แล้ว

`min-h-dvh` / `h-dvh` ใช้ครบทั้ง 8 จุด **ไม่มี `100vh` หรือ `min-h-screen` หลงเหลือ**
→ ปัญหาคลาสสิกที่ URL bar ของ Safari บังเนื้อหาไม่เกิดกับเรา

## ขอบเขต

1. ตั้ง `viewportFit: "cover"` → `env()` ที่เขียนไว้แล้วเริ่มทำงาน
2. พอเปิด `cover` เนื้อหาจะไหลไปใต้รอยบากตอนหมุนจอแนวนอน → กัน inset ซ้าย/ขวาที่ `body`
   (จุดเดียว ไม่ต้องไล่แก้ทุก container และไม่ชนกับ `lg:` ที่มีอยู่)
3. เพิ่ม e2e project ที่ 320px — ให้ CI จับแทนที่จะรอเจอบนเครื่องจริง
4. ลด gutter เหลือ `px-3` เฉพาะจอแคบกว่า 375px คืนพื้นที่ให้เนื้อหา 8px ต่อข้าง

## ข้อจำกัดที่ปิดไม่ได้

**ยืนยันบน iPhone จริงไม่ได้จากที่นี่** — Playwright รัน Chromium ไม่ใช่ WebKit บนอุปกรณ์จริง
ข้อ 1 ยืนยันได้จาก spec ของ CSS Env และ HTML ที่ server ส่งจริงเท่านั้น
**ต้องให้คีนเปิดบนเครื่องยืนยันอีกรอบหลัง deploy**

## Definition of done

- [x] HTML ที่ server ส่งมี `viewport-fit=cover`
- [x] เปิด e2e ที่ 320px แล้วเขียว
- [x] จอ ≥375px gutter ยังเป็น 16px เท่าเดิม · จอ <375px เป็น 12px
- [x] ด่าน CI ครบ 5 + `npm run e2e`

### 22 ก.ค. 2026 — A

**ยืนยันจาก HTML ที่ server ส่งจริง:**
`width=device-width, initial-scale=1, viewport-fit=cover` → `env()` ใน `app-nav.tsx`
ที่เขียนไว้ตั้งแต่แรกเริ่มทำงานแล้ว

**gutter วัดจริงต่อความกว้าง** (`main#main` padding-left):

| จอ | gutter | เนื้อหา |
| --- | --- | --- |
| 320 | 12px | 296px (เดิม 288) |
| 360 · 374 | 12px | 336 · 350 |
| **375** | **16px** | 343 |
| 390 · 430 | 16px | 358 · 398 |

ไม่มี h-scroll สักความกว้าง · จอ ≥375px ได้ค่าเดิมเป๊ะ ไม่มีอะไรเปลี่ยน

**safe-area ซ้าย/ขวา** วางที่ `body` จุดเดียว — ไม่ต้องไล่แก้ทุก container
และไม่ชนกับ `lg:px-10` ที่มีอยู่ · ในแนวตั้ง `env()` เป็น 0 อยู่แล้ว จึงไม่กระทบอะไรเลย
นอกจากตอนหมุนจอแนวนอนบนเครื่องที่มีรอยบาก

เพิ่ม `pt-[env(safe-area-inset-top)]` ที่ header มือถือด้วย — ใน Safari ปกติเป็น 0
แต่ถ้าใครกด Add to Home Screen จะรันโหมด standalone แล้วค่านี้มีผลทันที

### 🐛 e2e project ใหม่จับบั๊กได้ทันทีในการรันครั้งแรก

`/reflection` มี h-scroll 4px ที่ 320px — **และเป็นบั๊กที่ผมทำเองในคอมมิตนี้**

`week-picker.tsx` ใช้ pattern `-mx-4 ... px-4` เพื่อให้แถบเลื่อนแตะขอบจอ
ซึ่งสมมติว่า gutter ของหน้าเป็น 16px · พอผมลด gutter เหลือ 12px ที่จอแคบ
`-mx-4` เลยล้ำออกไปเกินข้างละ 4px

แก้เป็น `-mx-3 px-3 xs:-mx-4 xs:px-4` ให้ bleed ตรงกับ gutter ทั้งสองช่วง
ไล่ทั้งโปรเจกต์แล้วมีจุดนี้จุดเดียวที่ใช้ pattern นี้
